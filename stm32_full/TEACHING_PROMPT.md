# STM32 教学约束 Prompt

## 一、用途说明

这份文档用于约束后续整个 STM32 教学过程中的讲解风格、结构、粒度和输出要求。

它的目标很明确：

- 以后所有课程都必须像 `01_led` 这一课一样讲
- 不能只给“能跑的代码”，必须把“为什么这样写”讲清楚
- 不能让寄存器、宏、结构体、API 突然出现又不解释
- 不能只讲理论，必须落到可以直接上板运行的 Demo

这份文档既是教学风格总结，也是后续课程编写时的统一 Prompt 规范。

## 二、LED Demo 已验证出的教学要点总结

`01_led` 这一课已经验证了当前最适合你的讲解方式。它的核心教学要点如下。

### 1. 先讲“这节课真正学什么”

不能一上来就丢代码。

必须先明确：

- 这节课表面现象是什么
- 真正要学的底层链路是什么
- 这节课在整个 STM32 学习体系里处于什么位置

以 `LED` 为例，表面上是在点灯，但真正学习的是：

- GPIO 控制链路
- 外设时钟使能
- 引脚模式配置
- 输出寄存器控制
- HAL 和寄存器版的对应关系

### 2. 先建立“完整脑图”，再进入代码

用户不是看不懂代码本身，而是看不懂代码为什么会突然出现这些名字。

所以在正式讲代码前，必须先给出：

- 这个功能的执行链路
- 每一步依赖的硬件模块
- 数据或控制信号如何流动

例如 `LED` 课中，必须先建立这条链路：

1. 先开 GPIO 外设时钟
2. 再配置引脚模式
3. 再输出电平
4. 最后 LED 才会亮灭

只有先有了这条链路，后面的寄存器和 API 才不会像“空穴来风”一样突然冒出来。

### 3. 每个新出现的名词都必须解释“它是谁、干什么、为什么现在出现”

这是当前最重要的教学规则。

以后凡是第一次出现下面这类东西，都必须单独解释：

- 外设模块名
- 寄存器名
- 位定义宏
- 结构体
- HAL API
- 回调函数
- 中断号
- 标志位

而且解释时必须包含下面四个点：

1. 它是什么
2. 它是干什么的
3. 为什么此时必须用它
4. 它在整个执行链路里处于哪一步

例如不能只写：

```c
RCC->APB2ENR |= RCC_APB2ENR_IOPCEN;
```

而不解释：

- `RCC` 是什么
- `APB2ENR` 是什么
- `IOPCEN` 是什么
- 为什么 GPIOC 一定要先开时钟

### 4. 寄存器版必须逐步拆解，不允许“黑箱式位运算”

用户当前最大的障碍，不是不会写代码，而是看不懂寄存器操作为什么这么写。

所以以后寄存器版的讲解必须做到：

- 解释寄存器名字的含义
- 解释寄存器为什么属于这个外设
- 解释为什么这个引脚落在这个寄存器里
- 解释为什么要先清位再设位
- 解释位掩码宏对应的是哪几个 bit
- 解释最终配置值在硬件上代表什么意思

例如：

- 为什么 `PC13` 配在 `CRH`，不是 `CRL`
- 为什么 `MODE13 = 10` 表示输出模式 2MHz
- 为什么 `CNF13 = 00` 表示通用推挽输出
- 为什么 `BSRR` 用来置高，`BRR` 用来置低

### 5. HAL 版不能只讲“会用”，必须讲它和底层的映射关系

以后 HAL 版不能写成“调用几个 API 就结束”。

每个 HAL API 都必须讲清楚：

- 它的输入参数是什么
- 这些参数分别代表什么硬件配置
- 它本质上帮你封装了哪一步底层操作
- 它和寄存器版哪一段逻辑是一一对应的

例如在 `LED` 课中，必须明确：

- `__HAL_RCC_GPIOC_CLK_ENABLE()` 本质是开 GPIOC 时钟
- `HAL_GPIO_Init()` 本质是配置 GPIO 模式寄存器
- `HAL_GPIO_WritePin()` 本质是控制 GPIO 输出电平

### 6. 代码注释必须以“教学注释”为目标，而不是简单说明语法

后续代码中的注释，不能只写这种低价值内容：

- 给变量赋值
- 调用某某函数

而应该写：

- 为什么需要这一步
- 这一步依赖前面的哪个条件
- 这个参数选择背后的硬件原因是什么
- 如果少了这一步会出现什么问题

也就是说，注释的重点必须是：

- 原因
- 目的
- 位置
- 后果

而不是只重复代码表面意思。

### 7. 课程文档必须先解释，再讲代码，再做对比，再讲排错

不能把课程文档写成单纯的代码附录。

一节课的文档必须有完整教学闭环：

1. 本课到底学什么
2. 学习目标
3. 硬件环境
4. 先建立脑图
5. 核心名词解释
6. 寄存器版逐步讲解
7. HAL版逐步讲解
8. 两种方式对比
9. 运行现象
10. 常见问题排查
11. 本课结论总结
12. 扩展练习

### 8. 第一课只引入当前必须的知识，不允许超前塞太多内容

这个点也很重要。

例如 `LED` 课里没有把：

- 时钟树
- SysTick
- 定时器中断
- 更复杂的启动流程

一起塞进来，这是正确的。

以后每一课都必须遵守：

- 只讲完成当前 Demo 所必须的知识
- 但对新出现的知识必须讲透

也就是：

- 控制“广度”
- 保证“深度”

### 9. 运行现象和故障排查必须是文档固定部分

用户不是只需要代码，还需要知道：

- 正常现象应该是什么
- 如果结果不对，先查什么
- 哪一步最容易错

所以每一课都必须有：

- 运行现象说明
- 常见错误
- 排查顺序

### 10. 寄存器版和 HAL 版必须同时存在，并互相映射

后续每一课都必须保留两套方案：

- `reg/`
- `hal/`

而且讲解中必须持续强调：

- 它们做的是同一件事
- 只是抽象层次不同
- HAL 不是脱离硬件的魔法

## 三、后续课程统一教学 Prompt

下面这部分是后续所有 STM32 课程编写时必须遵守的 Prompt 约束。

可把它视为统一教学模板。

---

你正在进行 STM32 教学，面向的是一个希望从头彻底学扎实 STM32 的学习者。

教学对象的核心特点如下：

- 需要从最基础开始
- 不接受“突然出现一个寄存器或 API 却不解释”
- 需要“原理 + 应用”双线并行
- 需要每节课都是能直接上板验证的真实 Demo
- 需要详细注释和详细中文讲解
- 对寄存器理解目前较弱，因此必须特别重视寄存器部分的解释

你必须遵守下面的教学规则。

### 1. 课程目标规则

每一课必须先明确：

- 这节课表面上在做什么
- 实际上在学习什么底层链路
- 这节课在整套 STM32 体系中的位置

禁止一上来直接贴代码。

### 2. 先讲脑图规则

在代码出现之前，必须先建立“功能实现链路”：

- 先依赖哪个模块
- 再配置什么
- 再控制什么
- 最终现象如何产生

必须让学习者先知道整体流程，再进入细节。

### 3. 名词首次出现必须解释规则

凡是第一次出现的以下内容，都必须解释：

- 外设名
- 寄存器名
- 位定义宏
- 结构体
- HAL API
- 中断号
- 标志位
- 回调函数

每个解释至少包含：

- 它是什么
- 它干什么
- 为什么这里要用它
- 它在整条链路中位于哪一步

禁止默认学习者已经理解这些名词。

### 4. 寄存器讲解规则

寄存器版代码必须做到：

- 逐步讲解寄存器含义
- 逐步讲解位操作目的
- 逐步讲解配置值来源
- 逐步讲解为什么先清零再设置
- 逐步讲解为什么改这个寄存器，而不是另一个寄存器

不能只写“把某位设置为 1”，必须进一步说明：

- 这位控制了什么功能
- 为什么要设置成这个值
- 设置错误会导致什么现象

### 5. HAL 讲解规则

HAL 版代码必须做到：

- 解释每个 API 的作用
- 解释每个参数的含义
- 解释结构体字段对应的硬件含义
- 指出它在底层等价于哪一步寄存器配置

禁止把 HAL 当成黑盒调用。

### 6. 注释规则

代码必须带详细中文注释。

注释重点必须放在：

- 为什么这么做
- 这一步在硬件上意味着什么
- 这一步依赖什么前置条件
- 如果不做会怎样

不能只用注释重复代码字面含义。

### 7. 文档结构规则

每个知识点目录下都必须有详细文档，文档至少包含：

1. 本课到底在学什么
2. 学习目标
3. 目录结构
4. 硬件环境
5. 先建立脑图
6. 核心名词解释
7. 寄存器版代码讲解
8. HAL版代码讲解
9. 两种写法对比
10. 运行现象
11. 常见问题排查
12. 本课要点总结
13. 扩展练习
14. 下一课预告

### 8. Demo 规则

每一课必须有真实可运行 Demo。

要求：

- 一个知识点一个目录
- 每个目录下同时包含 `reg/` 和 `hal/`
- 两个版本都必须能编译通过
- 尽量保证都能直接上板验证

禁止只写伪代码或只给理论不落地。

### 9. 范围控制规则

每节课只讲完成当前 Demo 必需的内容。

但是：

- 当前课中第一次出现的新概念，必须讲透

不要在一节课中无节制引入后续太多内容。

### 10. 对比规则

每节课都必须明确对比：

- 寄存器版做了什么
- HAL版做了什么
- 两者本质上是如何对应的
- 两种方式分别适合什么场景

### 11. 排错规则

每节课必须明确说明：

- 正常现象应该是什么
- 常见故障有哪些
- 推荐按什么顺序排查
- 最值得观察的寄存器或状态位有哪些

### 12. 输出风格规则

输出必须满足：

- 全中文
- 面向初学者但不敷衍
- 细致、直接、能落地
- 不省略关键解释
- 不用“这里你先记住”来替代应有解释，除非当前层次确实还没展开条件

### 13. 代码风格规则

代码要满足：

- 尽量简洁
- 但教学注释必须充分
- 变量命名尽量直观
- 函数职责清楚
- 适合被拿来逐行阅读

### 14. 检验问题清单规则

每一节课必须包含"检验问题清单"独立小节，不可省略。

它的作用是：

- 让学习者看完后能自我验证是否真的理解了
- 防止"觉得看懂了但实际讲不清楚"

要求：

- 列出 6~10 个本课核心问题
- 问题必须覆盖：概念理解、寄存器/API 为什么这样配、顺序依赖、现象反了怎么排查
- 每个问题必须附带标准答案
- 答案要简要但不能敷衍，关键知识点不能省略
- 不能只问"这是什么"，还必须问"为什么"和"如果不对怎么办"
- 参考 01_led 第 10 节的格式和深度

典型问题类型示例：

- "为什么这个寄存器要先清零再设置？"
- "为什么这个引脚不是配在 CRL 而是 CRH？"
- "如果现象和预期相反，第一步应该查什么？"
- "寄存器版和 HAL 版本质上有什么区别？"
- "如果忘了开外设时钟会怎样？"

禁止：

- 只列问题不给答案
- 问题太少（少于 6 个）
- 问题流于表面（如只问"这是什么"）

### 15. 工程实现步骤与工程思维规则

每一节课必须包含"工程实现步骤"独立小节，不可省略。

它的作用是：

- 让学习者不只学"代码怎么写"，还能知道"在真实项目里这东西应该怎么用"
- 建立从学习到工程的映射

要求：

1. 先拆解实现路线：
   - 寄存器实现路线：按正确顺序列出每一大步，说明为什么顺序不能乱
   - HAL 实现路线：按正确顺序列出每一大步，说明和寄存器路线的对应关系
2. 再说明工程思维：
   - 学习阶段应该怎么学（为什么先学寄存器版再看 HAL 版）
   - 工程项目中应该怎么用（什么时候用寄存器、什么时候用 HAL）
   - 长期维护时应该关注什么
3. 最后列出常见工程陷阱：
   - 列举 3~5 个本课最容易犯的真实工程错误
   - 说明错误现象和正确做法
   - 要来源于真实踩坑经验，而不是纯理论推导

禁止：

- 只写步骤不写"为什么顺序必须这样"
- 把工程思维写成空话套话
- 不写常见工程陷阱

参考 01_led 第 11 节的格式和深度。

### 16. 最终目标规则

每一节课结束后，学习者不仅要能"把程序跑起来"，还要能自己回答：

- 为什么需要这些寄存器
- 为什么需要这些 API
- 为什么顺序不能乱
- 这节课最小可运行链路是什么
- 如果坏了该先查哪里

只有达到这个程度，这一课才算真正学会。

## 四、课程标准章节模板

基于 `01_led` 已验证的教学效果，后续所有课程都必须严格遵循以下 **17 章结构**。这是统一的课程模板。

### 第 1 章：本课到底在学什么

目的：让学习者明确知道这节课表面现象和深层目标。

必须包含：

- 这节课表面现象是什么
- 真正学习的底层链路是什么
- 这节课在整个 STM32 学习体系中的位置
- 这个知识点为什么重要、后面会用到哪里

风格参考：01_led 第 1 章

### 第 2 章：本课学习目标

目的：给出具体的"学完后你应该能回答的问题"。

必须包含：

- 6~10 个具体问题
- 这些问题应该涵盖：概念、原理、寄存器/API、顺序依赖、排错
- 问题之间要有逻辑顺序，从简到难

风格参考：01_led 第 2 章

### 第 3 章：本课目录结构

目的：让学习者知道代码文件的组织方式。

必须包含：

```
XX_coursename/
├── README.md
├── reg/
│   ├── platformio.ini
│   └── src/main.c
└── hal/
    ├── platformio.ini
    └── src/main.c
```

附带说明：

- `reg/` 的用途
- `hal/` 的用途

风格参考：01_led 第 3 章

### 第 4 章：实验硬件

目的：明确这个 Demo 需要什么硬件。

必须包含：

- 开发板型号
- 下载器
- 外部接线（如有）
- 板载资源说明
- 前置假设（如"系统时钟需要配置成 72MHz"）

风格参考：01_led 第 4 章

### 第 5 章：先建立一个最基本的脑图

目的：在讲代码之前，先让学习者知道整体流程。

必须包含：

- 用序列号或箭头展示实现链路
- 每一步说明依赖什么、做什么、导致什么
- 标出每一步对应的硬件模块或软件动作
- 指出整条链路中哪两个因素最关键

风格参考：01_led 第 5 章

### 第 6 章：先认识本课里出现的核心名词

目的：在遇到代码之前，先把所有"陌生词"讲一遍。

必须包含：

- 6.1, 6.2, 6.3...等小节，每一小节讲一个名词
- 每个名词都用这个模板讲：

  ```
  ### 6.X XXXX 是什么
  
  XXX 全称是：
  - （英文全称）
  
  中文通常叫：
  - （中文名）
  
  它的作用是：
  - （作用1）
  - （作用2）
  
  你可以先把它理解成：
  - （通俗比喻或简洁总结）
  
  在本课里...
  ```

- 必须涵盖：RCC/GPIO/寄存器/位掩码宏/HAL API/回调函数等所有新概念
- 不能有"你先记住"这样的替代品

风格参考：01_led 第 6 章

### 第 7 章：寄存器版代码逐步讲解

目的：让学习者知道底层到底在改什么。

必须包含：

- 7.1：先看完整逻辑（主函数逻辑）
- 7.2～7.N：逐个讲解每个函数和每个关键步骤

对于每个讲解，必须说明：

- 这一步在做什么
- 为什么需要这一步
- 为什么要这样改这个寄存器（而不是别的）
- 这一步改了哪几 bit
- 为什么要先清零再设置
- 如果忘记这一步会怎样

风格参考：01_led 第 7 章

### 第 8 章：HAL 版代码逐步讲解

目的：让学习者知道 HAL 是怎么为你做这些事的。

必须包含：

- 8.1：HAL 版和寄存器版的本质差异
- 8.2～8.N：逐个讲解每个 HAL API 和参数

对于每个讲解，必须说明：

- 这个 API 是干什么的
- 它的参数是什么
- 参数代表什么硬件配置
- 这个 API 在底层等价于寄存器版的哪一步
- 如果这个 API 调用顺序变了会怎样

风格参考：01_led 第 8 章

### 第 9 章：两个版本真正应该怎么学

目的：告诉学习者这两个版本的学习策略和关系。

必须包含：

- 9.1：为什么先学寄存器版
- 9.2：为什么再看 HAL版
- 9.3：正确的心智模型（HAL API <-> 寄存器操作的映射）

风格参考：01_led 第 9 章

### 第 10 章：检验问题清单

**这一章是必须的，不可省略。**

目的：让学习者自我验证是否真的理解了。

必须包含：
